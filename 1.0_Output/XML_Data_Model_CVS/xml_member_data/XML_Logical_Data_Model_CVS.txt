### 2. Logical Data Model

#### Person
This table contains demographic and personal information about an individual.

| Column Name      | Description                                 | Data Type |
| :--------------- | :------------------------------------------ | :-------- |
| DOB              | Date of birth of the person                 | DATE      |
| Gender           | Gender of the person                        | STRING    |
| Race             | Racial identity of the person               | STRING    |
| Ethnicity        | Ethnic background of the person             | STRING    |
| MaritalStatus    | Marital status of the person                | STRING    |
| EmploymentStatus | Current employment status                   | STRING    |
| EducationLevel   | Highest level of education attained         | STRING    |
| Nationality      | Nationality of the person                   | STRING    |
| EmergencyContact | Emergency contact details                   | STRING    |
| LastUpdated      | Last modification timestamp of the record   | TIMESTAMP |
| SourceSystem     | System where the data originated            | STRING    |
| IngestionTimestamp| Timestamp when the record was ingested      | TIMESTAMP |

#### ContactInfo
This table stores the contact details of an individual.

| Column Name   | Description                   | Data Type |
| :------------ | :---------------------------- | :-------- |
| Address       | Current address of the person | STRING    |
| ContactNumber | Contact number of the person  | STRING    |
| EmailId       | Email address of the person   | STRING    |

#### Language
This table stores the language preferences of an individual.

| Column Name | Description                | Data Type |
| :---------- | :------------------------- | :-------- |
| Spoken      | Primary spoken language    | STRING    |
| Written     | Primary written language   | STRING    |

#### HealthInfo
This table contains health-related information about an individual.

| Column Name | Description                            | Data Type |
| :---------- | :------------------------------------- | :-------- |
| Insurance   | Health insurance provider company name | STRING    |
| Provider    | Primary healthcare provider            | STRING    |

### 3. Relationships

*   **Person to ContactInfo:** One-to-One relationship. Each person has one set of contact information.
*   **Person to Language:** One-to-One relationship. Each person has one set of language preferences.
*   **Person to HealthInfo:** One-to-One relationship. Each person has one set of health information.

### 4. Rationale and Assumptions

*   **Normalization:** The data has been normalized into four tables: `Person`, `ContactInfo`, `Language`, and `HealthInfo`. This reduces data redundancy and improves data integrity. For example, a person's contact information, language, and health information are stored in separate tables, linked by a common person identifier.
*   **PII Classification:** Fields that can directly or indirectly identify an individual have been classified as PII. This is crucial for data privacy and compliance with regulations like GDPR and HIPAA.
*   **Data Types:** Data types have been chosen to be as specific as possible to ensure data quality. For example, `DOB` is a `DATE` and `LastUpdated` is a `TIMESTAMP`.
*   **Exclusion of Keys:** As per the instructions, primary and foreign keys are not included in the logical data model output. However, it is assumed that a `PersonID` would be used as a primary key in the `Person` table and as a foreign key in the other tables to establish the relationships.
*   **Assumptions:**
    *   It is assumed that the `PersonID` is a unique identifier for each person across all tables.
    *   The conceptual model provided is the single source of truth for the logical data model design.
    *   The provided XML data is a representative sample of the data that will be stored in the data platform.
    *   The `Gender` column in the conceptual model was an `INT` but has been changed to `STRING` in the logical model to be more descriptive and avoid ambiguity (e.g., 'Male', 'Female', 'Other').

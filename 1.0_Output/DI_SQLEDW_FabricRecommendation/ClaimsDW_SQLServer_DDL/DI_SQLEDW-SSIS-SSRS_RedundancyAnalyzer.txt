# SQL Server EDW Optimization Analysis  

## 1. Introduction  
This report presents a comprehensive analysis of the SQL Server Enterprise Data Warehouse (EDW) environment, focusing on the identification of redundant and unused data objects, and optimization opportunities. The analysis covers usage patterns within SSIS packages, SSRS RDL files, and the EDW schema, with the aim of improving efficiency, scalability, and usability. Optimizing the EDW is critical for ensuring high performance of ETL (SSIS) and reporting (SSRS) processes, reducing maintenance overhead, and supporting the organization’s data-driven decision-making.

## 2. Usage Analysis  

### SSIS Packages  
- **Tables Used as Targets:**  
  - `DimClaim`, `DimCheck`, `DimClaimant`, `DimExposure`, `DimLossLocation`, `DimUser`, `FactClaimTransaction`, `LossBodyParts` are all used as targets in their respective SSIS packages (e.g., `EDW_CC_Load_DimClaim.dtsx`).
- **Tables Used as Lookups:**  
  - Each dimension and fact table uses `PublicID`, `BeanVersion`, and key columns for lookup operations in their respective packages.
- **Unused Tables:**  
  - `dmproc.Audit`, `dmproc.AuditReference`, `dmproc.ErrorLog`, `dmproc.BatchIdentifier` are not used as targets or lookups in any SSIS package.

### SSRS RDL Files  
- **Tables and Columns Referenced:**  
  - `DimClaim`, `FactClaimTransaction`, `DimUser` are referenced in multiple reports (e.g., Claims Financial Performance, Claims Operational Efficiency, ClaimsStatusSummary).
  - Key columns used include: `ClaimID`, `ClaimStatus`, `ClaimCreateDate`, `CloseDate`, `ClaimType`, `IsActive`, `AssignmentStatusName`, `TransactionCreateDate`, `IndemnityPaid`, `MedicalPaid`, `EmployersLiabilityPaid`, `ExpensePaid`, `IndemnityReserve`, `MedicalReserve`, `EmployerLiabilityReserve`, `ExpenseReserve`, `IndemnityRecovery`, `MedicalRecovery`, `EmployerLiabilityRecovery`, `ExpenseRecovery`, `SubrogationRecovery`, `UserName`, `FirstName`, `LastName`.
- **Unreferenced Tables:**  
  - `DimCheck`, `DimClaimant`, `DimExposure`, `DimLossLocation`, `LossBodyParts`, all `dmproc.*` tables are not referenced in any SSRS report.

## 4. Recommendations  

### EDW Data Model Optimization  
- **Remove or Archive Unused Tables and Columns:**  
  - Tables and columns not referenced in any SSIS or SSRS process (e.g., `dmproc.Audit`, `dmproc.AuditReference`, `dmproc.ErrorLog`, `dmproc.BatchIdentifier`, and many columns in dimensions and fact tables) should be reviewed for deprecation or archival.
- **Document Data Lineage:**  
  - Update documentation to reflect actual usage and lineage, supporting ongoing governance and future optimization.
- **Review and Rationalize Dimension Attributes:**  
  - For dimensions with many unused attributes, validate with business stakeholders before removal.

### SSIS Pipeline Design Improvements  
- **Consolidate ELT Logic:**  
  - Where the same table and columns are used as targets in multiple packages (e.g., SCD logic for `DimClaim`, `DimCheck`, etc.), consolidate logic into shared or parameterized packages to reduce redundancy and maintenance.
- **Optimize Lookups:**  
  - Centralize and cache lookup tables to improve ETL performance, especially for frequently used dimension keys.
- **Batch Processing and Error Handling:**  
  - Standardize batch and error handling logic across all packages for consistency and easier troubleshooting.

### SSRS Report Design Refinements  
- **Remove Unused Data Elements:**  
  - Simplify datasets by removing unused columns, improving report performance and clarity.
- **Enhance Data Model for Reporting:**  
  - Where possible, denormalize or pre-aggregate data for high-usage reports to reduce runtime computation.
- **Improve Documentation:**  
  - Clearly document report data sources and calculations for maintainability.

## 5. Conclusion  
This analysis has identified key optimization opportunities in the SQL Server EDW environment. By removing or archiving unused tables and columns, consolidating redundant SSIS package logic, and refining SSRS report designs, the EDW can be made more efficient, scalable, and maintainable. Implementing these recommendations will enhance data pipeline performance, reduce storage and maintenance costs, and improve reporting clarity, supporting the organization’s ongoing data-driven initiatives.
